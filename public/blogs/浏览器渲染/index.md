## 前言
> 浏览器渲染原理是前端性能优化的核心基石，我们常说的重排、重绘、GPU 加速等优化手段，本质上都是基于浏览器渲染流水线的底层逻辑。从网络拿到 HTML/CSS/JS 资源，到屏幕上呈现完整页面，浏览器渲染引擎会经过 5 个环环相扣的核心阶段。本文结合ai和视频学习将逐层拆解每个阶段的工作原理、核心逻辑、关键特性，以及对应的前端开发优化要点。

------

## 一、阶段 1：HTML 解析与 DOM 树构建
![](https://www.runoob.com/wp-content/uploads/2013/09/ct_htmltree.gif)
### 1.1 核心目标
将网络请求返回的 HTML 二进制字节流，转换为浏览器可识别、可操作的**DOM 树（文档对象模型）**。DOM 树是页面内容的结构化载体，也是后续所有渲染阶段的基础输入。

### 1.2 完整解析流程

1. **字节转字符**：浏览器接收网络返回的 HTML 二进制字节流，通过指定编码格式（如 UTF-8）转换为可读的文本字符串。
2. **词法分析（Token 化）**：通过分词器将文本字符串拆解为符合 HTML 规范的 Token，分为标签 Token（开始标签、结束标签，如`<html>`、`</div>`）和内容 Token（文本、注释等）。
3. **语法分析（节点生成）**：根据 HTML 语法规则，将 Token 转换为对应的 DOM 节点，同时解析节点的属性、事件绑定等元信息。
4. **DOM 树构建**：根据 Token 的层级嵌套关系，将 DOM 节点组织成树形结构。根节点为`document.documentElement`（`<html>`标签），子节点按照 HTML 的嵌套顺序依次挂载，完整还原页面的内容结构与层级关系。

### 1.3 关键特性与优化要点

- **增量解析**：浏览器无需等待完整 HTML 下载完成，即可边接收边解析，提前启动后续渲染流程。
- **解析阻塞规则**：遇到`<script>`标签时，HTML 解析会默认暂停，等待 JS 文件下载并执行完成（JS 可修改 DOM 结构，浏览器必须保证 DOM 树的一致性）；可通过`async`/`defer`属性解除阻塞，`async`下载完成后立即执行，`defer`等待 HTML 解析完成后按顺序执行。
- **预解析扫描器**：HTML 解析暂停时，浏览器会启动预扫描器，提前扫描后续的外链资源（JS、CSS、图片等）并发起下载请求，减少整体加载耗时。

------

## 二、阶段 2：CSS 解析与样式计算（Style）
### 2.1 核心目标

解析所有 CSS 样式资源，生成**CSSOM 树（CSS 对象模型）**，并为 DOM 树中的每个可见节点，计算出最终的样式属性值，生成`ComputedStyle`（计算样式），为后续布局阶段提供精准的样式依据。

### 2.2 核心工作流程

1. **CSS 资源解析与 CSSOM 树构建**

   - 浏览器会全量收集所有来源的 CSS 样式：用户代理样式（浏览器默认样式）、行内样式、`<style>`内嵌样式、`<link>`外链样式。
   - 与 HTML 解析流程一致，CSS 字节流会经过字符转换、Token 化、语法分析，最终生成树形结构的 CSSOM 树。CSSOM 树与 DOM 树结构一一对应，完整保留了每个选择器的样式规则、继承关系与层叠关系。
   - 核心特性：**CSSOM 的构建会阻塞页面渲染**，没有完整的样式信息，浏览器无法确定元素的最终呈现效果，因此 CSS 被称为「阻塞渲染的资源」，建议将 CSS 外链放在`<head>`中尽早加载解析。

   

2. **样式计算与 ComputedStyle 生成**

   这是本阶段的核心，浏览器会为每个 DOM 节点匹配对应的样式规则，计算出最终的绝对样式值，分为 3 个核心步骤：

   

   - **规则匹配**：为 DOM 节点匹配所有命中的 CSS 选择器规则，关键细节：**CSS 选择器匹配是从右往左进行的**，比如`div .content p`，浏览器会先匹配所有`p`标签，再向上验证父级是否符合规则，大幅提升匹配效率。
   - **层叠优先级计算**：按照 CSS 层叠规范，对同一节点的多个样式规则进行优先级排序，生效优先级为：`!important` > 行内样式 > ID 选择器 > 类 / 伪类 / 属性选择器 > 标签选择器 > 浏览器默认样式。
   - **绝对值转换**：将样式中的相对单位（`em`/`rem`/`%`/`vw`）、关键字（`bold`/`red`）转换为浏览器可直接计算的绝对值，比如`font-size: 2rem`转换为对应`px`值，`color: red`转换为 RGBA 值，最终生成每个节点的`ComputedStyle`。

   

### 2.3 优化要点

- 避免过深的 CSS 选择器嵌套，减少样式匹配的计算开销；
- 减少`!important`的滥用，避免样式层叠逻辑混乱与维护成本提升；
- 内联首屏关键 CSS，非首屏 CSS 采用异步加载，减少 CSSOM 构建对首屏渲染的阻塞。

## 三、阶段 3：布局（Layout / 重排）

### 3.1 核心目标

有了 DOM 树和每个节点的 ComputedStyle，浏览器仍无法确定元素在页面中的精确位置和尺寸。布局阶段（业内也叫回流）的核心，就是生成**布局树（Layout Tree，也叫渲染树 Render Tree）**，并计算出所有可见元素的几何属性（精确的坐标位置、宽高尺寸），为后续绘制阶段提供空间坐标依据。

### 3.2 核心工作流程

1. **布局树（Render Tree）构建**

   布局树只包含页面中**可见的节点**，会过滤掉不可见节点：

   

   - 非可视化标签：`<head>`、`<meta>`、`<script>`等标签不会进入布局树；

   - 样式隐藏节点：设置了`display: none`的节点及其子节点，不会进入布局树；

   - 注意：

     ```
     visibility: hidden
     ```

     ```
     opacity: 0
     ```

     的节点，仅视觉上不可见，仍占据页面空间，会被纳入布局树。

     

     布局树的每个节点，都会关联对应的 DOM 节点和 ComputedStyle，完整保留元素的内容和样式信息。

   

2. **几何布局计算**

   浏览器会根据 CSS 布局模型（标准流、浮动、Flex、Grid、定位等），从根节点开始递归计算每个节点的精确几何属性：

   

   - 基于盒模型，计算元素的内容宽高、内边距、边框、外边距；
   - 基于包含块（Containing Block）规则，计算元素的相对 / 绝对坐标位置；
   - 处理文本换行、滚动区域、浮动元素位置避让等复杂场景。

   

### 3.3 关键概念：重排（回流）

重排指元素的几何属性发生变化时，浏览器需要重新触发布局计算，更新布局树，再重新执行后续的绘制、合成流程。

- 重排的核心触发场景

  ：

  1. DOM 节点的增减、移动、替换；
  2. 元素的宽高、位置、边距、边框等几何属性修改；
  3. 页面窗口大小变化（`resize`事件）；
  4. 主动获取元素的实时几何属性：`offsetTop`/`offsetLeft`/`clientWidth`/`getBoundingClientRect()`等，浏览器为了获取最新值，会强制触发布局计算。

  

- **核心特性**：**重排的开销极高**，一个元素的布局变化，可能会连锁触发其父级、子级、兄弟节点的布局重新计算，是前端页面卡顿的核心元凶之一。

### 3.4 优化要点

- 避免频繁修改元素的几何属性，尽量批量修改 DOM 样式，比如使用`class`替代多次`style`修改；
- 对频繁触发重排的元素，开启绝对定位使其脱离文档流，避免连锁触发整个页面的重排；
- 避免频繁读取`offsetTop`等实时几何属性，读取操作尽量集中，避免读写交替触发强制同步布局。

------

## 四、阶段 4：绘制（Paint / 重绘）

### 4.1 核心目标

布局阶段已经确定了元素的位置、尺寸等几何信息，绘制阶段的核心，就是将元素的视觉内容（颜色、背景、阴影、边框、文本、图片等），转换为一系列的**绘制指令列表**，并按照堆叠顺序，为每个图层生成对应的像素位图，为最终的屏幕呈现提供像素数据。

### 4.2 核心工作流程

1. **绘制分层（Paint Layers）**

   浏览器不会把整个页面生成一个绘制指令列表，而是会按照规则，将布局树拆分为多个独立的**绘制层**，每个层独立生成绘制指令，避免局部修改触发整个页面的重绘。

   

   - 常见分层规则：根元素（文档层）、设置了`position: absolute/fixed`的元素、形成堆叠上下文的元素、`canvas`/`video`等媒体元素、设置了`overflow: scroll`的元素等。

   

2. **绘制指令生成与执行**

   浏览器会为每个绘制层，生成对应的绘制指令列表，指令顺序遵循 CSS 的堆叠上下文规则，和手绘逻辑一致：先绘制背景、再绘制边框、再绘制内容、最后绘制上层元素。

   例如绘制一个带背景、边框、文本的 div，会生成如下核心指令：

   

   1. 绘制背景色（`background-color`）；
   2. 绘制背景图片（`background-image`）；
   3. 绘制边框（`border`）；
   4. 绘制内部文本内容；
   5. 绘制内阴影、外阴影（`box-shadow`）。

   

### 4.3 关键概念：重绘

重绘指元素的视觉属性发生变化，但不影响其几何位置和布局时，浏览器无需重新执行布局阶段，只需要重新生成该元素所在图层的绘制指令，重新执行像素绘制的过程。

- **重绘的核心触发场景**：修改元素的`color`、`background-color`、`box-shadow`、`border-radius`、`visibility`等不影响布局的视觉属性。
- **核心特性**：重绘的开销远小于重排，因为它跳过了布局计算阶段，但仍有像素绘制的开销；**重排一定会触发重绘，重绘不一定触发重排**。

### 4.4 优化要点

- 避免频繁修改视觉属性，减少重绘的触发频率；
- 对频繁重绘的元素，提升为独立的绘制层，避免局部重绘触发整个图层的重绘；
- 减少复杂的`box-shadow`、渐变等效果，这类绘制的像素计算开销极高。

------

## 五、阶段 5：合成（Composite）

### 5.1 核心目标

合成是浏览器渲染流水线的最后一个阶段，核心目标是将前面绘制好的多个绘制层，按照正确的堆叠顺序，合并为一个完整的位图，最终交给 GPU 渲染到屏幕上，呈现出用户看到的页面。同时，合成阶段也是浏览器实现 GPU 加速、流畅动画的核心环节。

### 5.2 核心工作流程

1. **合成层提升**

   浏览器会将符合规则的绘制层，进一步提升为**独立的合成层**，每个合成层都会拥有独立的 GPU 纹理（位图），GPU 可以直接对这些纹理进行处理，无需浏览器主线程参与。

   

   - 常见的合成层提升规则：

     1. 设置了 3D/2D `transform`属性的元素；
     2. 设置了`will-change: transform/opacity`的元素；
     3. `video`、`canvas`、`webgl`等硬件加速的媒体元素；
     4. 正在执行`opacity`动画、CSS 滤镜的元素；
     5. 被其他合成层覆盖的元素，会被自动提升为独立合成层。

     

   

2. **图层合成与 GPU 渲染**

   

   - 主线程将所有合成层的位图、层级顺序、动画信息，提交给**合成线程**；
   - 合成线程对图层进行裁剪、视口处理（比如页面滚动时，只处理可视区域的图层），将图层的位图上传到 GPU 作为纹理；
   - GPU 根据合成线程的指令，将多个纹理按照正确的顺序进行合并、混合，最终渲染到屏幕的帧缓冲区，完成整个页面的渲染。

   

### 5.3 核心优势与关键特性

1. **极致的性能提升**：合成阶段的操作完全在合成线程和 GPU 中执行，**不会占用浏览器主线程**。也就是说，即使主线程被 JS 阻塞，合成线程控制的动画依然可以流畅执行。
2. **跳过重排重绘**：修改合成层的`transform`、`opacity`属性时，不会触发布局和绘制阶段，只会触发合成阶段的图层合并，这就是我们常说的「GPU 加速」，也是前端实现流畅动画的最优方案。
3. **离屏渲染与滚动优化**：页面滚动时，合成线程可以提前预渲染可视区域外的内容，同时对滚动条、固定定位元素单独处理，实现流畅的滚动体验。

### 5.4 优化要点与避坑指南

- 动画优先使用`transform`和`opacity`实现，替代`top`/`left`/`width`等会触发重排的属性，替代`background-color`等会触发重绘的属性；
- 对即将执行动画的元素，提前设置`will-change: transform`，让浏览器提前将其提升为合成层，做好动画准备，避免动画开始时的卡顿；
- **避免过度使用合成层**：每个合成层都会占用 GPU 的显存，过多的合成层会导致内存占用飙升，反而造成性能下降、页面闪烁等问题；
- 避免合成层的层级重叠，防止出现不必要的层提升，造成内存浪费。

------

## 总结

浏览器的渲染流水线，是一个「内容解析→样式计算→空间布局→像素绘制→图层合成」的完整闭环，5 个阶段环环相扣，前一个阶段的输出，是后一个阶段的输入。

- 首屏优化：聚焦 DOM/CSSOM 的构建速度，减少渲染阻塞；
- 运行时性能优化：尽量减少重排、重绘的触发，优先使用合成层实现交互与动画，把开销极高的主线程操作，转移给 GPU 高效处理。
